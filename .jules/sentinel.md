## 2024-05-15 - Unbounded JSON Payload DoS and Missing Type Check
**Vulnerability:** The HTTP server in `gui.py` parsed `Content-Length` and directly read that number of bytes from the request stream without any limit check. This enables a denial-of-service (DoS) attack where a large payload causes resource exhaustion. In addition, the parsed JSON body `payload` was assumed to be a dictionary, leading to `AttributeError` exceptions (e.g. `payload.get()` on lists, strings, integers) when handling malicious or malformed input types. The unhandled exception crashed the request handler.
**Learning:** Even internal or local-only GUI servers should protect against unexpected input sizes and enforce rigid type checks on JSON schemas. Relying on `json.loads` alone does not validate the type of the top-level entity, meaning defensive type validation (`isinstance(payload, dict)`) is strictly necessary before accessing object attributes or dictionary methods.
**Prevention:** Always implement a `MAX_INPUT_LENGTH` boundary check before calling stream read methods (`.read()`) on incoming user data. After deserializing JSON payloads, explicitly validate that the resulting object matches the expected structure and Python type (e.g., using `isinstance()` or a schema validator).
